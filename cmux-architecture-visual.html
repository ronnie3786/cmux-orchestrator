How commands flow from you → through the system → into Claude Code sessions

Dashboard UI (localhost:7779) — Two views:

HTTP API (POST /api/*)
Layer 2 — Harness Server (Python)
🧠 HarnessEngine engine.py
Main loop thread. Polls workspaces every 5s. Manages:
• Session lifecycle (start/complete/idle detection)
• Auto-approval of permission prompts
• Review pipeline (Sonnet/Haiku/Ollama)
• Screen cache + fingerprinting
• Houses the Orchestrator instance
🎯 Orchestrator orchestrator.py
The "Tech Lead" brain. When you send a goal:
1. Creates Planner session (Claude Code)
2. Planner outputs task plan → parsed by Haiku
3. Spawns Worker sessions per task
4. Monitors, reviews, reports back
Uses _append_message() for chat history
📋 Planner planner.py
Builds planning prompts, parses task output via Haiku.
Output: structured task list with dependencies, files, checkpoints.
👷 Worker worker.py
Creates git worktrees per task.
Each worker = isolated branch + directory.
orchestrator/{task-id}-{slug}
📡 Monitor monitor.py
Watches session screens for completion signals, errors, stuck states.
cmux Socket API (JSON-RPC v2)
Layer 3 — cmux (Multiplexer)
🔌 cmux Socket cmux_api.py
Unix socket at ~/Library/.../cmux.sock
Key operations the harness uses:
• workspace.list — enumerate sessions
• workspace.new — create new workspace
• surface.read_text — read terminal screen
• surface.send_text — type text into terminal
• surface.send_key — send Enter, Ctrl-C, etc.
• workspace.close — kill workspace
📨 send_prompt_to_workspace() The Injection Point
This is how ALL text gets into Claude Code:

1. Try tmux paste-buffer (atomic, handles long text)
2. Fallback: surface.send_text + surface.send_key("enter")

Every orchestrator task, every planner prompt, every user message — all funnels through this function.
tmux pane / terminal surface

Command Flow: Today vs. Action Buttons

Where do action buttons fit in this stack?

✅ Current: Orchestrator Chat

1
You type a goal in the Orchestrator chat panel
2
Orchestrator receives via POST /api/orchestrator/message
3
Orchestrator creates a Planner workspace via cmux API
4
send_prompt_to_workspace() injects goal into Planner's Claude Code
5
Orchestrator parses plan → spawns Worker sessions → same injection path
6
Claude Code sessions execute, monitored by engine loop

⚡ Proposed: Action Button (Direct)

1
You click "Build and Run" FAB button
2
UI sends POST /api/inject-action with saved prompt text + target workspace
3
send_prompt_to_workspace() injects prompt directly into the target Claude Code session
4
Claude Code executes the skill/command. Done.
—
No planner, no task decomposition, no orchestrator brain. Just a direct prompt injection into an existing session.

🎯 The Key Question: Where to Inject?

Three options, depending on the action button's intent

Option A: Existing Workspace Direct
Inject into whichever Claude Code session is currently active for the project.

Best for: /exp-project-run, quick commands, skill invocations
Target: The active workspace's Claude Code REPL
Path: FAB → send_prompt_to_workspace(existing_uuid, prompt)

⚠️ Requires an active Claude Code session to inject into
Option B: New Session Spawn
Create a fresh Claude Code workspace, then inject the prompt.

Best for: "Build and Run" where you want a clean session at the project worktree
Target: New workspace via workspace.new → then inject
Path: FAB → create workspace → send_prompt_to_workspace(new_uuid, prompt)

✅ Always works, no dependency on existing session state
Option C: Orchestrator Orchestrated
Feed the prompt through the orchestrator as if it were a chat message / goal.

Best for: Complex multi-step actions that benefit from planning
Target: Orchestrator's message handler
Path: FAB → POST /api/orchestrator/message → full planning flow

🧠 Gets planner decomposition, worker isolation, reviews
💡 Recommendation
Most action buttons should be Option A or B (direct injection). The whole point is speed: one click, done.

Each button could store a mode field: "inject" (into active session), "spawn" (new session), or "orchestrate" (full planning).

"Build and Run" = Option B (spawn a fresh session at project dir, inject /exp-project-run)
"Run Tests" = Option A (inject into active session)
"Refactor module X" = Option C (let orchestrator plan it out)

🧠 Harness Server — Python Backend
The brain: engine, orchestrator, planner, workers, monitor
server.py ▼
⚙️ HarnessEngine engine.py
The main loop. A daemon thread polling every 5 seconds.

• Reads all workspace screens via cmux
• Detects Claude Code sessions (active/idle/exited)
• Auto-approves permission prompts
• Triggers reviews on session completion
• Tracks costs, fingerprints, session IDs
• Owns the Orchestrator instance
🎯 Orchestrator orchestrator.py
The "Tech Lead" brain. 1,900+ lines of orchestration logic.

• Receives goals from chat UI
• Creates Planner session → gets task plan
• Parses plan via Haiku (cheapest LLM)
• Launches workers respecting dependencies
• Reviews results, manages rework cycles
• Runs its own Claude Code session for Q&A
• Idle sweep closes stale sessions after 1hr
📋 Planner planner.py
Goal → Task decomposition.

• Builds structured planning prompt
• Rules: max parallelism, no chains >2 deep
• Small tasks (1-3 files each), no overlap
• Output: plan.md in worktree
• Haiku parses plan → JSON task queue
• 3-6 tasks per objective (sweet spot)
👷 Worker worker.py
Task execution setup.

• Creates git worktree per objective (shared branch)
• Writes spec.md + context.md to worktree
• Launches Claude Code workspace at worktree path
• Prompt: "Read spec.md, implement, write result.md"
• Context from completed deps piped to next worker
📡 Monitor monitor.py
Session health tracking.

• Checks if Claude Code is still running
• Detects stuck/errored sessions
• Reads terminal screen for progress signals
• Triggers completion flow when session ends
🔍 Review review.py
Code quality gate.

• Runs claude --print on completed diffs
• Sonnet reviews catch real bugs
• Can trigger rework cycles (up to N retries)
• Results displayed in task cards

🔌 cmux — Terminal Multiplexer
The bridge between your code and Claude Code's terminal
Unix Socket ▼
📡 Socket API cmux_api.py
The single transport layer. Every interaction with Claude Code goes through here.

Socket: ~/Library/.../cmux.sock

Key operations:
• workspace.create — new terminal workspace
• workspace.list — enumerate all workspaces
• workspace.rename — set workspace title
• workspace.close — kill workspace
• surface.read_text — read terminal screen
• surface.send_text — type text into terminal
• surface.send_key — send Enter, Ctrl-C, etc.
⭐ send_prompt_to_workspace() THE FUNNEL
Every single prompt enters Claude Code through this function.

1. Try tmux paste-buffer (atomic, handles long text)
2. Fallback: surface.send_text() + surface.send_key("enter")

Orchestrator goals, planner prompts, worker task specs, user messages, skill commands — all of them funnel through this one function. This is where action buttons would inject too.

Orchestrator Goal Flow

What happens when you type "Fix the auth token refresh bug" in the orchestrator chat

Layer 1 — Dashboard UI
You type a goal in the orchestrator chat panel
"Fix the auth token refresh bug in IOSDOX-24739"
Layer 2 — Server
Server receives via POST /api/orchestrator/message
Creates an Objective with goal, project dir, and creates a git worktree for isolation.
Layer 2 — Orchestrator
Orchestrator starts planning phase
Spawns a background thread _run_planning(). Status → "planning".
Layer 3 — cmux
Creates Planner workspace via workspace.create
New terminal workspace. Runs cd /worktree && claude. Waits for REPL ready (❯ prompt).
Layer 3 — cmux
send_prompt_to_workspace() injects planning prompt
The structured prompt with rules: maximize parallelism, small tasks, explicit file ownership. Tells Claude to write plan.md.
Layer 4 — Claude Code
Planner session analyzes codebase
Claude reads relevant files, understands the problem, writes a task plan to plan.md. Auto-approved for file writes.
Layer 2 — Orchestrator
Detects plan.md exists → parses with Haiku
Polls every 10s for plan.md. Once found, sends to claude --print --model haiku to extract structured JSON: tasks, files, dependencies. Status → "plan_review".
Layer 1 — Dashboard UI
Plan displayed for your review
Task cards appear with titles, files, dependencies. You approve or edit. Click "Execute" to proceed.
Layer 2 — Orchestrator
Launches ready tasks (respecting dependencies)
_launch_ready_tasks(): picks first queued task with all deps met. Writes spec.md + context.md to worktree. One worker at a time.
Layer 3 → Layer 4
Worker workspace created, task prompt injected
Same path: workspace.create → cd worktree && claude → wait for REPL → send_prompt_to_workspace() with "Read spec.md, implement the task".
Layer 4 — Claude Code
Worker implements the task
Claude reads spec, makes code changes, writes result.md with summary of what was done.
Layer 2 — Review
Sonnet reviews the changes
claude --print runs a diff review. If issues found → rework cycle (new prompt, same worktree). If clean → task complete, next task launches.
Layer 1 — Dashboard
All tasks done → Objective complete
Status → "completed". Orchestrator chat session stays open for Q&A. You can review diffs, ask questions, merge when ready.

What happens when you type directly in a Command Center workspace

Server receives via POST /api/send
Payload: workspace index/UUID + text. Server validates and forwards.
Layer 3 — cmux

Same function as orchestrator uses. tmux paste-buffer or send_text + send_key("enter").
Layer 4 — Claude Code
Claude Code processes your input

Layer 2 — Engine
Engine detects completion → triggers review
5s polling loop detects session exit. Captures snapshot, runs review if enabled. Results shown in dashboard.

Action Button Injection — Finalized Design

Always spawn a new session. Orchestrator keeps full visibility. Never blocked.

🚀
Spawn New Session in Worktree DECIDED
Every action button spawns a fresh Claude Code workspace. Orchestrator monitors it like any worker.
🔘 Click FAB → _create_worker_workspace() → cd worktree && claude → wait for REPL ❯ → send_prompt_to_workspace() → Claude executes
✅ Why This Works
• Orchestrator never blocked — stays free for Q&A
• Full visibility — task registered on objective, shows in task cards
• Engine auto-monitors — approval, screen reads, completion detection
• Review pipeline — Sonnet reviews on completion, same as planned work
• Reuses existing code — _create_worker_workspace(), _wait_for_repl(), send_prompt_to_workspace()
💬 Orchestrator Awareness
You: "How's the build going?"

Orchestrator reads the action-button task's workspace screen via cmux, sees the live terminal output, and reports:

"Build is at 4/7 modules compiled. ~3 min remaining based on current pace."

This works because the task is registered on the objective with a workspaceId the orchestrator can read.
📦 Task Registration
Action-button sessions get added to the objective's task array with "source": "action-button" so they're distinguishable from planned tasks but tracked identically:

{ "id": "action-build-run-1712249040", "title": "Build & Run", "source": "action-button", "status": "executing", "workspaceId": "uuid", "prompt": "/exp-project-run" }

📨
send_prompt_to_workspace()
cmux_api.py:226
THE funnel point. Every prompt enters Claude Code through this. Tries tmux paste-buffer first (atomic), falls back to send_text + send_key("enter"). Never embeds newlines in send_text.
📡
Monitor
monitor.py
Watches session screens for completion signals. detect_claude_session() checks if Claude Code is still active. Triggers the completion → review pipeline when sessions end.
🔍
Review System
review.py
Code quality gate. Uses claude --print (Sonnet) to review diffs. Can trigger rework cycles. Backends: Sonnet (best), Haiku (fast), Ollama (free/local).
🌐
Dashboard Server
server.py
HTTP server on port 7779. Serves dashboard.html + orchestrator.html. REST API for all UI interactions. Routes to Engine and Orchestrator methods.
📦
Objective
objectives.py
A high-level goal + its task queue. Stored as JSON in ~/.cmux-harness/objectives/[uuid]/. Has status lifecycle: planning → plan_review → executing → completed/failed.
🔒
WorkspaceMutex
workspace_mutex.py
Prevents race conditions when multiple parts of the system try to send keys/text to the same workspace simultaneously. Uses cooldown timers per workspace UUID.
✅
Auto-Approval
approval.py + engine.py
Detects Claude Code permission prompts ("Do you want to...") and auto-sends Enter. Configurable per workspace. Also handles trust-folder and compact-mode prompts during REPL startup.
